# Codex CLI: Token Usage Format Spec

This document records the on-disk format, token fields, and session boundary
heuristics for the Codex CLI session log. It is the reference for implementing
`CodexProvider` in annulus (#118b).

---

## On-disk path

Codex resolves the session log root in this order:

1. `$CODEX_HOME/sessions/` — if the `CODEX_HOME` environment variable is set.
2. `~/.codex/sessions/` — the default when `CODEX_HOME` is absent.

Within that root, sessions are stored as JSONL files under a date hierarchy:

```
~/.codex/sessions/YYYY/MM/DD/<session-id>.jsonl
```

Older sessions are moved to a flat archive directory:

```
~/.codex/archived_sessions/<session-id>.jsonl
```

The annulus stub currently probes `~/.codex/usage.json` (a rate-limit snapshot
written by older CLI versions). The real per-turn token data lives in the
`sessions/` tree documented above. The `usage.json` probe can coexist; it
remains useful for `is_available()` detection.

### Per-platform paths

| Platform | Default path |
|----------|--------------|
| macOS    | `~/.codex/sessions/YYYY/MM/DD/<id>.jsonl` |
| Linux    | `~/.codex/sessions/YYYY/MM/DD/<id>.jsonl` |
| Windows  | `%USERPROFILE%\.codex\sessions\YYYY\MM\DD\<id>.jsonl` — not directly observed; inferred from the `dirs::home_dir()` behavior used by the Rust CLI. TODO: verify on a real Windows install. |

`CODEX_HOME` overrides the base directory on all platforms.

## Format

Each session file is **NDJSON** (one JSON object per line, newline-delimited).
Lines are appended as events arrive; a file may be written to by an active Codex
session while annulus reads it.

There are several entry `type` values. The three that matter for token
accounting are:

### `session_meta` — session context

Appears once at the start of a session. Carries the session ID, working
directory, and optional model-provider metadata.

```json
{
  "timestamp": "2025-09-11T18:25:00.000Z",
  "type": "session_meta",
  "payload": {
    "id": "9f8e7d6c-5b4a-3210-fedc-ba9876543210",
    "cwd": "/home/user/myproject",
    "model_provider": "openai"
  }
}
```

### `turn_context` — model marker per turn

Appears before each assistant turn. Carries the model name used for that turn.
This is the primary source for associating subsequent token counts with a model.

```json
{
  "timestamp": "2025-09-11T18:25:30.000Z",
  "type": "turn_context",
  "payload": {
    "model": "gpt-5",
    "cwd": "/home/user/myproject"
  }
}
```

### `event_msg` with `payload.type == "token_count"` — token accounting entry

The primary token-usage record. May appear multiple times per turn. Contains
cumulative totals for the whole session so far (`total_token_usage`) and
optionally the delta for the most recent request (`last_token_usage`).

```json
{
  "timestamp": "2025-09-11T18:25:40.670Z",
  "type": "event_msg",
  "payload": {
    "type": "token_count",
    "info": {
      "model": "gpt-5",
      "total_token_usage": {
        "input_tokens": 1200,
        "cached_input_tokens": 200,
        "output_tokens": 500,
        "reasoning_output_tokens": 0,
        "total_tokens": 1700
      },
      "last_token_usage": {
        "input_tokens": 1200,
        "cached_input_tokens": 200,
        "output_tokens": 500,
        "reasoning_output_tokens": 0,
        "total_tokens": 1700
      }
    }
  }
}
```

Other `event_msg` subtypes (`user_message`, `agent_message`, etc.) exist but
carry no token counts and can be skipped.

## Token fields

All token counts are non-negative integers. Field names are as they appear
inside `info.total_token_usage` and `info.last_token_usage`.

| Field | Meaning | Billing treatment |
|-------|---------|-------------------|
| `input_tokens` | Prompt tokens sent to the model this turn | Priced at input rate minus cached share |
| `cached_input_tokens` | Subset of `input_tokens` satisfied from the prompt cache (`cache_read_input_tokens` is an alias seen in some versions) | Priced at the cheaper cached-input rate (falls back to input price when missing from the pricing table) |
| `output_tokens` | Completion tokens generated by the model; **includes reasoning tokens** | Priced at output rate |
| `reasoning_output_tokens` | Breakdown of the reasoning portion already counted in `output_tokens` | Informational only — do not add to the billable total again |
| `total_tokens` | Provided by modern Codex builds; equals `input + output` (reasoning is not double-counted) | Use verbatim when present |

### Mapping to `TokenUsage`

```
TokenUsage::prompt_tokens       <- input_tokens
TokenUsage::completion_tokens   <- output_tokens
TokenUsage::cache_read_tokens   <- cached_input_tokens (or cache_read_input_tokens)
TokenUsage::cache_creation_tokens <- not emitted by Codex; leave as 0
```

### Delta vs. cumulative

- `last_token_usage` — per-turn delta. Preferred when present.
- `total_token_usage` — running cumulative total for the whole session. Use to
  recover a per-turn delta by subtracting the previous cumulative reading.
- Legacy builds (pre-September 2025) may omit `last_token_usage`; synthesize the
  delta with subtraction.
- When neither field is present, skip the entry.

### Legacy `total_tokens` synthesis

When `total_tokens` is absent from the raw record, compute:
`total_tokens = input_tokens + output_tokens`

Do not add `reasoning_output_tokens` again — it is already included in
`output_tokens`.

## Session boundary

Codex does not write an explicit end-of-session marker. Session boundaries are
structural and temporal.

### Structural boundary

Each JSONL file represents exactly one session. The filename encodes the session
ID (the same value that appears in `session_meta.payload.id`). Reading a single
file gives you the token totals for that session; no cross-file accumulation is
needed for per-session aggregation.

### Recency heuristic (for `last_session_at`)

To identify the most recent session, compare the filesystem `mtime` of JSONL
files in `~/.codex/sessions/` (and `archived_sessions/`). The file with the
newest `mtime` corresponds to the most recent session. Alternatively, take the
`timestamp` field of the last line in that file.

### Date hierarchy

The path `sessions/YYYY/MM/DD/<id>.jsonl` encodes the calendar date of the
session. Use the directory components to scope queries to a time range without
reading every file.

## Edge cases

The reader must tolerate:

1. **Partial writes** — the last line of an active session file may be
   incomplete JSON. Skip lines that fail to parse; do not abort the whole file.
2. **Missing `model` metadata** — early Codex builds (pre-September 2025) do
   not emit `turn_context` or `info.model`. Fall back to the last known model,
   or flag events as `isFallbackModel = true` (the ccusage Codex CLI uses
   `gpt-5` as the fallback). Annulus can leave the model name empty or use a
   sentinel.
3. **Both `last_token_usage` and `total_token_usage` absent** — skip the
   `token_count` entry entirely.
4. **Concurrent sessions** — multiple Codex instances may write to different
   files simultaneously. Treat each file independently; there is no cross-file
   lock to acquire.
5. **Log rotation / archiving** — completed sessions move to
   `archived_sessions/` as flat files. The reader must scan both directories to
   cover the full history.
6. **`cache_read_input_tokens` alias** — some builds emit this field name
   instead of `cached_input_tokens`. Accept both; prefer
   `cached_input_tokens` when both are present.
