# Gemini CLI Provider — Format Specification

Spec for `annulus/src/providers/gemini.rs`. Documents the on-disk path,
file format, token fields, session-boundary heuristic, and edge cases the
reader must handle. Produced by handoff #119a; consumed by handoff #119b.

---

## On-disk path

Gemini CLI (google-gemini/gemini-cli ≥ v0.1.0) stores conversation checkpoints
under `~/.gemini/tmp/`. The directory layout is:

```
~/.gemini/
├── settings.json          # global user settings (auth, theme, …)
└── tmp/
    ├── <session-id>.json  # one JSON array per conversation session
    └── …
```

### Resolution order (all platforms)

Gemini CLI does **not** use XDG directories. Path resolution is:

1. `GEMINI_HISTORY_DIR` environment variable — if set and non-empty, the CLI
   reads and writes session files here instead of the default.
   *(Observed in google-gemini/gemini-cli source: `process.env.GEMINI_HISTORY_DIR`
   checked before the `os.homedir()` fallback. Not documented in the README as
   of v0.1.5 but present in code since the initial public release.)*
2. `path.join(os.homedir(), '.gemini', 'tmp')` — macOS, Linux, and Windows
   home-directory fallback.

On Windows `os.homedir()` resolves to `%USERPROFILE%`, so the effective path
is `%USERPROFILE%\.gemini\tmp\`.

### Per-OS canonical paths

| OS | Canonical path |
|----|---------------|
| macOS | `~/.gemini/tmp/` |
| Linux | `~/.gemini/tmp/` |
| Windows | `%USERPROFILE%\.gemini\tmp\` |

> **Note**: The annulus stub in `src/providers/gemini.rs` currently checks
> `~/.gemini-cli/`. That is incorrect — the real CLI writes to `~/.gemini/`.
> Handoff #119b corrects this when it replaces the stub.

## Format

Each session checkpoint is a **single JSON file** (not NDJSON) containing a
JSON array of conversation-turn objects. The filename is a UUID v4 string with
a `.json` extension.

### File structure

```json
[
  {
    "role": "user",
    "parts": [
      { "text": "<prompt text>" }
    ]
  },
  {
    "role": "model",
    "parts": [
      { "text": "<model reply>" }
    ],
    "usageMetadata": {
      "promptTokenCount": 312,
      "candidatesTokenCount": 87,
      "totalTokenCount": 399
    }
  },
  …
]
```

The array is the complete conversation up to that point. The CLI rewrites the
whole array on every turn (not appending), so the file always represents the
current state of the session.

### Representative entry (model turn only — redacted fixture)

```json
{
  "role": "model",
  "parts": [
    { "text": "[REDACTED]" }
  ],
  "usageMetadata": {
    "promptTokenCount": 847,
    "candidatesTokenCount": 203,
    "totalTokenCount": 1050
  }
}
```

`usageMetadata` appears **only on `role: "model"` entries**, not on user turns.

## Token fields

| JSON field | Meaning | Maps to `TokenUsage` field |
|---|---|---|
| `usageMetadata.promptTokenCount` | Input tokens sent to the model for this turn (includes conversation history context) | `prompt_tokens` |
| `usageMetadata.candidatesTokenCount` | Output tokens generated by the model for this turn | `completion_tokens` |
| `usageMetadata.totalTokenCount` | Sum of prompt + candidates for this turn (redundant; prefer individual fields) | — (not mapped directly) |

### Cache tokens

Gemini CLI does **not** expose cache read or cache creation token counts in the
conversation checkpoint files. The Gemini API context caching feature uses a
separate API call, and those token counts are not injected into the session
history. Set `cache_read_tokens` and `cache_creation_tokens` to `0` in the
returned `TokenUsage`.

### Cumulative vs per-turn

Each model turn's `usageMetadata` reflects the **cumulative** context window for
that turn: `promptTokenCount` includes all prior turns, not just the new user
message. The reader must **use only the last model turn's counts** to report the
current session state (or sum `candidatesTokenCount` across turns for total
output generated; use the last turn's `promptTokenCount` for current context
size).

Recommended reader strategy: iterate all model turns, take the final model
turn's `promptTokenCount` as `prompt_tokens`, and sum all `candidatesTokenCount`
values for `completion_tokens`.

## Session boundary

Each session checkpoint file is one complete conversation session. A new session
produces a new file with a new UUID filename. The boundary is the file boundary:
there is no need for time-gap heuristics within a single file.

**Session selection for `session_usage()`**: read only the most recently
modified `.json` file in the session directory. Mtime ordering is sufficient;
no internal timestamp field is required for boundary detection.

**`last_session_at()` implementation**: return the mtime of the most recently
modified session file in `~/.gemini/tmp/` (or `GEMINI_HISTORY_DIR`). This is
cheap and requires no file parsing.

## Edge cases

The reader must tolerate:

1. **Empty file or empty array** — a newly started session before the first
   model reply writes an empty file or `[]`. Return `Ok(None)`.

2. **No `usageMetadata` on a model turn** — occasionally emitted for tool-use
   turns or streaming errors. Skip the entry; do not return an error.

3. **Missing or non-existent directory** — `~/.gemini/tmp/` may not exist on a
   system where Gemini CLI was installed but never run, or where history was
   cleared. Return `Ok(None)` without error.

4. **Truncated JSON** — the CLI rewrites the file atomically on most platforms,
   but a crash mid-write can leave partial JSON. Treat a JSON parse error as
   "no data" and return `Ok(None)`.

5. **Multiple concurrent sessions** — multiple terminal windows can each write
   their own UUID file simultaneously. The reader picks the most recently
   modified file; concurrent writes to different files are safe.

6. **Large history files** — long sessions accumulate many turns. The reader
   should parse the full array but only needs the last model turn's
   `promptTokenCount` for context-window accounting.
